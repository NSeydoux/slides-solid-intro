An overview of OIDC stakeholders

The Resource Owner

  • (Fancy name for a user)
  • Controls the data.
  • Uses applications to get services.

The Client

  • (Fancy name for an application)
  • Consumes the data.
  • Creates more data.
  • Receives delagation by the Resource Owner.

The Resource Server

  • Fancy name for a Pod.
  • It hosts the data.
  • It enforces access permissions.

The OIDC Provider

  • It is trusted by the Resource Owner.
  • It is trusted by the Resource Server.
  • This is where the Resource Owner account is.

Overview of the OIDC "dance"

RO requests login to Client

  • The Client doesn't manage the RO identity
  • The Client dedirects the RO to the OP
  • Some information about the client is sent to the OP

RO logs in OP

  • The RO authenticates to the OP.
  • The authentication method is out of scope.
  • The Client never sees the RO credentials.

OP gets RO consent

  • The Client announces to OP
    what access it requires:
    • RO WebID
    • Permission to access data when RO is offline
  • The RO can approve or deny.
  • OP asks only once.

OP redirects RO to Client

  • The OP provides the Client with a single-use code.
  • The OP asociates the code to the Client.

Client sends code

  • The Client sends the code to the OP
    along with its information.
  • (This is transparent to the RO.)

OP sends tokens

  • The ID token is meant for the Client.
  • The Access Token is meant for the RS.

Authenticated access

  • The Client can access private resources.
  • The OP lets the RO revoke the Access Token.

What does Solid-OIDC add?

What does Solid-OIDC add?

Universal user identity

The RO has a WebID

  • The user identity is an IRI: the WebID
  • This IRI points to a profile document
  • The profile doc enables discovering:
    • The RO's trusted OP
    • The RO's Pod root

OP signals WebID to Client

  • The ID token contains the WebID
  • The Client can discover the RO's profile

What does Solid-OIDC add?

Universal Client identity

OIDC security model

  • The Client controls some IRI known to the OP
  • Tokens are only sent to this trusted IRI

In OIDC:
Static client registration

  • The developer registers Client to the OP.
  • The obtained Client credentials are long-lasting.
  • Suitable for a server-side Client

In OIDC:
Dynamic client registration

  • Client information is provided at runtime
  • De facto anonymous Client: anybody can get identifiers
  • The Client credentials are ephemeral
  • Suitable for browser-side apps

In Solid-OIDC:
Client Identifiers

  • No manual registration to the OP:
    Client information is discovered from Client Identifier
  • The Client ID is Client-controlled

OP signals Client Identifier

  • The ID token also includes the Client Identifier
